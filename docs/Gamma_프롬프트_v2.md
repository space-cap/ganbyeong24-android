## 슬라이드 구성 (10개 카드)

### 1. 표지
- 제목: 간병24
- 부제: 믿을 수 있는 간병 매칭 서비스
- 파란색 계열 디자인 (#2196F3)
- 발표자 정보 및 날짜

### 2. 목차
Part 1 내용:
1. 프로젝트 소개
2. 문제 정의 및 해결 방안
3. 주요 기능 (보호자/간병사/관리자)

Part 2 내용 (다음 프레젠테이션):
5. 기술 스택 및 아키텍처
6. 성과 및 향후 계획

### 3. 프로젝트 소개
- 무엇을: 간병인 매칭 플랫폼 Android 앱
- 왜: 복잡한 절차 없이 간편하게 간병 서비스 이용
- 누구를 위해: 40-70대 보호자 및 40-60대 간병사
- 핵심 가치 4가지:
  * 간편한 사용 (최소한의 입력)
  * 시니어 친화적 UI (큰 폰트, 명확한 버튼)
  * 빠른 매칭 (전화 상담 연결)
  * 안전한 데이터 관리 (Firebase)

### 4. 문제 정의
현재 간병 서비스의 문제점:
- 복잡한 절차: 기존 서비스는 회원가입 및 인증 절차가 복잡
- 낮은 접근성: 시니어 사용자가 사용하기 어려운 UI
- 느린 매칭: 간병사를 찾기까지 시간이 오래 걸림
- 신뢰성 부족: 간병사 정보 확인이 어려움

### 5. 해결 방안
간병24의 솔루션:
- 간편한 입력: 최소한의 필수 정보만 입력
- 시니어 친화 UI: 큰 폰트(20sp), 명확한 버튼(56dp)
- 전화 상담 매칭: 관리자가 직접 전화로 상담
- Firebase 기반 관리: 안전한 클라우드 데이터베이스

### 6. 사용자 역할 시스템
3가지 역할:
- 👥 보호자: 간병 신청, 내역 조회
- 👨‍⚕️ 간병사: 프로필 등록, 자격증 관리
- 👨‍💼 관리자: 신청 관리, 간병사 관리, 매칭 관리

### 7. 보호자 기능 - 간병 신청
3단계 프로세스:
1. 환자 정보 입력
   - 이름, 나이, 성별
   - 병원 및 병실 정보
2. 간병 조건 설정
   - 시작일, 기간
   - 특이사항 입력
3. 신청 완료
   - 자동 일련번호 생성: 100-0000-0001
   - 관리자 전화 상담 대기

### 8. 간병사 기능 - 등록
3단계 프로세스:
1. 프로필 정보
   - 사진 업로드
   - 이름, 성별, 연락처
2. 경력 및 자격
   - 경력 선택 (드롭다운)
   - 자격증 다중 선택 (칩)
   - 가능 지역 선택
3. 등록 완료
   - 자동 일련번호 생성: 200-0000-0001
   - 관리자 승인 대기

### 9. 관리자 시스템
3가지 관리 기능:
- 간병 신청 관리
  * 전체 조회
  * 상태별 필터링 (대기중/매칭완료/취소)
- 간병사 관리
  * 전체 조회
  * 지역/자격증/경력 필터링
- 매칭 관리
  * 3단계 매칭 프로세스
  * 신청 선택 → 간병사 선택 → 매칭 생성

### 10. UI/UX 디자인 원칙
시니어 친화적 디자인:
- 📱 큰 터치 영역: 최소 48dp x 48dp
- 🔤 큰 폰트: 본문 16sp 이상, 버튼 20sp
- 🎨 높은 대비: WCAG AA 기준 (4.5:1)
- 🔘 명확한 버튼: 높이 56dp, 둥근 모서리
- 📝 간단한 입력: 최소한의 필수 정보만
- 색상: 파란색 (#2196F3) 메인, 초록색 (#4CAF50) 포인트

## 디자인 요청사항

## 슬라이드 구성 (10개 카드)

### 1. Part 2 시작 (전환 슬라이드)
- 제목: 기술 스택 및 성과
- 부제: 간병24의 기술적 구현과 프로젝트 성과
- Part 1 요약: 시니어 친화적 간병 매칭 플랫폼

### 2. 기술 스택
- 언어: Kotlin 100%
- UI: Jetpack Compose, Material Design 3
- 아키텍처: MVVM (ViewModel + StateFlow)
- DI: Koin
- 비동기: Kotlin Coroutines
- 백엔드: Firebase (Auth, Firestore)
- 로컬 DB: Room Database
- 이미지: Coil

### 3. 아키텍처 다이어그램
MVVM 3계층 구조:
- Presentation Layer
  * Jetpack Compose UI
  * ViewModel
- Domain Layer
  * Repository
- Data Layer
  * Firebase Firestore
  * Room Database
  * Firebase Auth

### 4. 데이터베이스 구조
Firestore Collections:
- users: 사용자 정보 (role 포함)
- care_requests: 간병 신청 정보 (SerialNumber: 100-0000-0001)
- caregivers: 간병사 정보 (SerialNumber: 200-0000-0001)
- matches: 매칭 정보 (SerialNumber: 300-0000-0001)
- counters: SerialNumber 카운터

특징:
- Transaction 기반 일련번호 생성 (중복 방지)
- Soft Delete 구현 (데이터 보존)
- 실시간 동기화

### 5. 핵심 기술 - SerialNumber 시스템
문제:
- Firebase 자동 생성 ID는 사람이 읽기 어려움
- 고객 지원 시 식별이 어려움

해결:
- 사람이 읽기 쉬운 일련번호 체계 (XXX-XXXX-XXXX)
- 간병 신청: 100-0000-0001
- 간병사: 200-0000-0001
- 매칭: 300-0000-0001

장점:
- 타입 구분 가능 (첫 자리로 구분)
- 전화 상담 시 편리
- Transaction으로 중복 방지

### 7. 기술적 성취
완료된 기능:
- ✅ Firebase Authentication 연동
- ✅ MVVM 아키텍처 구현
- ✅ Koin DI 적용
- ✅ SerialNumber 시스템 구축
- ✅ 관리자 시스템 (3단계 매칭)
- ✅ Room Database 통합
- ✅ Soft Delete 구현

코드 품질:
- 100% Kotlin
- MVVM 패턴 준수

- Coroutines 비동기 처리

### 8. 배운 점 및 어려웠던 점
기술적 학습:
- Jetpack Compose의 선언형 UI 패턴
- Firebase Firestore의 실시간 동기화
- MVVM 아키텍처의 실전 적용
- Koin을 통한 의존성 주입
- Transaction 기반 데이터 일관성 보장

해결한 문제:
- SerialNumber 중복 생성 → Firestore Transaction 사용
- 복잡한 상태 관리 → StateFlow + sealed class 패턴
- 시니어 친화 UI 설계 → 접근성 가이드라인 준수
- Firebase 비동기 처리 → Kotlin Coroutines + Flow
